## what a UX portfolio must prove

when hiring managers look at a junior portfolio, they are asking:

1. **Do u have a process?** u didnt just make pretty screens. u figured out what to make and why.
2. **Can u make decisions?** u chose this layout, flow, or copy for a reason u can explain.
3. **Can u communicate?** a non-designer can follow the case study and understand what u did.

the #1 portfolio mistake is a Behance page full of polished mockups with no context.
beautiful screens with no research, no problem framing, and no trade-off explanation show execution, not design thinking.
execution might get a UI freelance gig.
decision-making and process get a full-time Product Designer role.

## what a case study must contain

each portfolio piece should tell this story:

1. **The problem** - what user need or business goal was this solving? where did the problem come from?
2. **Your role** - what were u responsible for? be specific and honest.
3. **The research** - what did u find out, and how? show at least one artifact.
4. **The design decisions** - show at least two versions of a key screen and explain why one won.
5. **The outcome** - usability test results, a metric, stakeholder sign-off, or what u learned.
6. **What youd do next** - what would iteration 2 address?

## case study types

aim for **3-4 case studies** across a few types.

| Type | What it demonstrates | Notes |
|---|---|---|
| **End-to-end product redesign** | full process from research to hi-fi | best centerpiece. pick a real app or website with real problems. |
| **0->1 new feature** | discovery, ideation, scoping | choose a genuine unmet need, not a feature that already exists. |
| **Usability audit + fix** | evaluative research + iteration | credible if the analysis is sharp. pair with a Stark accessibility audit. |
| **Accessibility-focused redesign** | WCAG 2.2 knowledge + systematic thinking | rare in junior portfolios, so it stands out. |
| **Design system or component** | systems thinking and consistency at scale | strongest for Product Designer and UI tracks. show rationale, not just output. |

## what gets rejected vs what gets hired

**rejected:**

- [ ] fake problems, like "i redesigned Airbnb for fun," with no user data
- [ ] process theater: double-diamond diagram with no actual research artifacts
- [ ] UI-only-for-UX: beautiful mockup, no case study, no problem, no process
- [ ] template sameness: community UI kit + pastel palette + lorem ipsum
- [ ] case studies where nothing was tested, nothing changed, and the first design was final
- [ ] portfolios where every project looks identical

**what gets hired:**

- [ ] real problem found through real research
- [ ] evidence of iteration after testing
- [ ] accessibility audit or compliance check done systematically
- [ ] research artifact with actual data: quotes, affinity map, usability clip
- [ ] honest caveats, like "given more time, id tree-test the navigation"
- [ ] realistic scope: a 3-screen feature with depth beats a 40-screen app with no depth

## Daily UI runs in parallel

[Daily UI](https://www.dailyui.co) is a 100-day email challenge with one prompt per day.
run it alongside your main case studies.

use it to:

- [ ] build craft velocity in Figma
- [ ] explore visual directions outside the main project
- [ ] produce shots for Behance or Dribbble

keep Daily UI separate from portfolio case studies.
Daily UI produces **shots**.
case studies show **process**.
your portfolio can use both, but label them clearly.

## the accessibility audit project

this is worth doing as a standalone case study.
its rare enough in junior portfolios to be a real differentiator.

**brief:** pick a real public app or website.
run a systematic WCAG 2.2 AA audit.
propose redesigns for the failures.

**deliverables:**

- [ ] Stark accessibility audit report for the existing UI
- [ ] annotated Figma screens showing each failure
- [ ] proposed redesigns for each failure with rationale
- [ ] written summary: severity, user impact, remediation priority

this project shows WCAG 2.2 knowledge, tool fluency, systematic thinking, and inclusive design.
that combination stands out meow.

## portfolio site

your portfolio needs a home.
options in order of effort:

| Option | What u get | Cost |
|---|---|---|
| **Behance** | free, indexed by Adobe, case-study friendly | **Free** |
| **Notion** | good for long-form case studies, link from a short bio page | **Free** |
| **Framer** (hobby tier) | real portfolio site with custom domain option and hi-fi prototypes | Free (watermarked) or ~$10/mo |
| **Custom built** | full control, only worth it if u can build it cleanly | Free-$10/mo |

Dribbble is for shots, not case studies.
a good Dribbble presence can build social proof, but it does not replace the case-study portfolio.
dont pay for Pro until u have shots worth sharing.

## visibility checklist

- [ ] 3-4 case studies live on your portfolio site
- [ ] each case study has: problem -> research -> decision -> outcome
- [ ] at least one case study includes real research, not made-up users
- [ ] at least one case study shows iteration
- [ ] one accessibility-focused piece, audit or redesign
- [ ] Daily UI shots live on Behance or Dribbble to show craft
- [ ] portfolio site loads fast, works on mobile, and has a working contact method

## exit check

- [ ] 3 case studies published with full problem-research-decisions-outcome narrative
- [ ] at least one case study includes real usability test data
- [ ] at least one case study shows a design decision that changed after testing
- [ ] one accessibility audit or compliance-focused case study
- [ ] portfolio site live with a clear introduction and contact method
- [ ] Daily UI Days 1-30+ completed and shots published
